# RHEL AI Midstream Versioning Guide

This guide documents how to implement RHEL AI midstream versioning (`X.Y.Z+rhaiv.N`) in a Python project using setuptools-scm and git tags.

## Overview

RHEL AI midstream versioning pattern: `$version+rhaiv.$build[.$accelerator]`

- **Base version**: Semantic version (e.g., `0.2.0`)
- **Midstream marker**: `+rhaiv` (Red Hat AI version identifier)
- **Build number**: `.N` (increments for midstream releases)
- **Accelerator** (optional): `.cuda`, `.rocm`, etc. (if applicable)

**Examples:**
- `0.2.0+rhaiv.1` - First RHEL AI midstream release of 0.2.0
- `0.2.0+rhaiv.2` - Second RHEL AI midstream release (e.g., security fix)
- `0.3.0+rhaiv.1` - New upstream 0.3.0, build counter resets to 1

---

## Step 1: Configure pyproject.toml

### 1.1 Update `[project]` section

```toml
[project]
name = "your-package-name"
dynamic = ["version"]  # Let setuptools-scm manage version
# ... other project metadata
```

### 1.2 Update `[build-system]` section

```toml
[build-system]
requires = ["setuptools>=80.9.0", "setuptools-scm[simple]>=9.2"]
build-backend = "setuptools.build_meta"
```

### 1.3 Add `[tool.setuptools_scm]` section

```toml
[tool.setuptools_scm]
# Automatically derive version from git tags
# Supports the midstream pattern: X.Y.Z+rhaiv.N
version_file = "src/your_package/_version.py"
# Preserve local version from tags (e.g., +rhaiv.N)
local_scheme = "no-local-version"
# Don't modify version - use tag as-is
version_scheme = "no-guess-dev"
```

**Important:** Replace `src/your_package/_version.py` with your actual package path.

---

## Step 2: Create Version File Loader

Create `src/your_package/__init__.py` (or update existing):

```python
"""Your package description."""

try:
    from ._version import __version__
except ImportError:
    # Fallback if _version.py doesn't exist (e.g., in dev mode without build)
    __version__ = "0.0.0+dev"

__all__ = ["__version__"]
```

**Note:** The `_version.py` file is auto-generated by setuptools-scm during build. Don't create it manually.

---

## Step 3: Update .gitignore

Add the auto-generated version file to `.gitignore`:

```gitignore
# setuptools-scm auto-generated version file
src/your_package/_version.py
```

---

## Step 4: Create and Push Version Tags

### 4.1 Tag Format

Tags must follow the pattern: `v{version}` or `{version}`

Examples:
- `v0.2.0+rhaiv.1`
- `0.2.0+rhaiv.1`

### 4.2 Create Initial Tag

```bash
# On main branch after merge
git checkout main
git pull origin main

# Create annotated tag
git tag -a v0.2.0+rhaiv.1 -m "Release 0.2.0+rhaiv.1"

# Push tag to remote
git push origin v0.2.0+rhaiv.1
```

### 4.3 Create Subsequent Tags

**For fixes/patches (increment build number):**
```bash
# Current: v0.2.0+rhaiv.1
# Next fix release:
git tag -a v0.2.0+rhaiv.2 -m "Release 0.2.0+rhaiv.2 - Security fixes"
git push origin v0.2.0+rhaiv.2
```

**For new upstream versions (reset build counter):**
```bash
# Current: v0.2.0+rhaiv.5
# New upstream version:
git tag -a v0.3.0+rhaiv.1 -m "Release 0.3.0+rhaiv.1 - New upstream version"
git push origin v0.3.0+rhaiv.1
```

## Step 6: Document in README.md

Add a versioning section to your `README.md`:

```markdown
## Versioning

This project follows the **RHEL AI midstream versioning pattern** for releases:

```
$version+rhaiv.$build[.$accelerator]
```

### Version Format

- **Base version**: Semantic version (e.g., `0.2.0`)
- **Midstream marker**: `+rhaiv` (Red Hat AI version)
- **Build number**: `.N` (increments for midstream releases)
- **Accelerator** (optional): `.cuda`, `.rocm`, etc; not applicable for this project

### Examples

| Version | Description |
|---------|-------------|
| `0.2.0+rhaiv.1` | First RHEL AI midstream release of 0.2.0 |
| `0.2.0+rhaiv.2` | Second RHEL AI midstream release (e.g., security fix) |
| `0.3.0+rhaiv.1` | New upstream 0.3.0, build counter resets to 1 |

### Installing Specific Versions

```bash
# Install latest version
pip install your-package

# Install specific midstream version
pip install your-package==0.2.0+rhaiv.1

# Check installed version
python -c "import your_package; print(your_package.__version__)"
```

### Version Progression Rules

1. **Tags are immutable**: Never delete, move, or replace existing tags
2. **Build counter increments**: Fix releases increment the build number (e.g., `+rhaiv.1` → `+rhaiv.2`)
3. **Counter resets on upstream bump**: New upstream version resets build to 1 (e.g., `0.2.0+rhaiv.5` → `0.3.0+rhaiv.1`)
4. **PEP 440 compliant**: Versions sort correctly using Python's version comparison
```

## Step 7: Containerfile Integration (Optional)

If building container images, integrate version into image labels:

```dockerfile
# Optional build arg for version label
ARG VERSION=""

# ... your build stages ...

# Image metadata
LABEL org.opencontainers.image.title="Your Package" \
      org.opencontainers.image.version="${VERSION}"
```

Build command:
```bash
# Auto-detect version from git
VERSION=$(git describe --tags --always)
podman build --build-arg VERSION=${VERSION} -t your-image:${VERSION} .
```

## Verification

### Verify Version Detection

```bash
# In development (without tags)
python -c "import your_package; print(your_package.__version__)"
# Output: 0.0.0+dev (fallback)

# After creating tag v0.2.0+rhaiv.1
git tag -a v0.2.0+rhaiv.1 -m "Release 0.2.0+rhaiv.1"

# Build and install
pip install -e .
python -c "import your_package; print(your_package.__version__)"
# Output: 0.2.0+rhaiv.1
```

### Verify in Built Package

```bash
# Build distribution
python -m build

# Check version in built wheel
unzip -p dist/your_package-*.whl your_package/_version.py
# Should show: __version__ = "0.2.0+rhaiv.1"
```

## Version Progression Examples

### Scenario 1: Fix Release

```bash
# Current state: v0.2.0+rhaiv.1 released
# Fix a bug, commit changes

git checkout main
git pull
# ... merge fix PR ...

# Create new tag (increment build)
git tag -a v0.2.0+rhaiv.2 -m "Release 0.2.0+rhaiv.2 - Bug fixes"
git push origin v0.2.0+rhaiv.2
```

### Scenario 2: New Upstream Version

```bash
# Current state: v0.2.0+rhaiv.5 released
# Bump to new upstream version 0.3.0

git checkout main
git pull
# ... merge upstream sync PR ...

# Create new tag (reset build counter)
git tag -a v0.3.0+rhaiv.1 -m "Release 0.3.0+rhaiv.1 - New upstream version"
git push origin v0.3.0+rhaiv.1
```

### Scenario 3: Security Update

```bash
# Current state: v0.2.0+rhaiv.3 released
# Apply security patch

git checkout main

# Create new tag (increment build)
git tag -a v0.2.0+rhaiv.4 -m "Release 0.2.0+rhaiv.4 - Security update"

## Important Rules

### DO ✅

- **Always use annotated tags**: `git tag -a v0.2.0+rhaiv.1 -m "message"`
- **Never delete or move tags**: Tags are immutable once pushed
- **Increment build number for fixes**: Within same base version
- **Reset build counter**: When bumping upstream version
- **Follow PEP 440**: Ensures proper version sorting
- **Document in CHANGELOG**: Every version release

### DON'T ❌

- **Don't use lightweight tags**: `git tag v0.2.0+rhaiv.1` (missing `-a`)
- **Don't delete pushed tags**: Breaks reproducibility
- **Don't skip build numbers**: Always increment sequentially
- **Don't manually edit _version.py**: It's auto-generated
- **Don't commit _version.py**: It's in .gitignore

## Troubleshooting

### Version shows as "0.0.0+dev"

**Cause:** No git tags exist or setuptools-scm can't find them.

**Solution:**
```bash
# Check if tags exist
git tag -l

# Ensure you're in a git repository
git status

# Create a tag
git tag -a v0.2.0+rhaiv.1 -m "Initial release"
```

### Version file not generated

**Cause:** Package not installed or built.

**Solution:**
```bash
# Install in editable mode
pip install -e .

# Or build the package
python -m build
```

### Version doesn't include +rhaiv suffix

**Cause:** Tag doesn't include the suffix.

**Solution:**
```bash
# Check tag format
git describe --tags

# Create properly formatted tag
git tag -a v0.2.0+rhaiv.1 -m "Release 0.2.0+rhaiv.1"
```
